## Goal

Let an operator **clone an existing session** as the starting
point for a new one — bring across the setup (reviewers,
reviewees, relationships, instruments, RTDs, RuleSets, email
template overrides, settings) **without** carrying responses,
audit history, or session-runtime state.

Today's only "carry over" path is **Settings CSV
import** (Segment 12A-3): operator exports `*_settings.csv`
from session A and uploads it into session B. That covers the
schema-shaped settings (instruments, RTDs, RuleSets, display
fields, email templates) but **not** the rosters (reviewers
/ reviewees / relationships). Cloning is the natural
"one-click" version.

## Why now

- `spec/operator_ui_concept.md` "Out of scope" §3 names
  **session duplication (sans response data)** as one of the
  "adjacent capabilities likely to land sooner" — it composes
  cleanly with the Overview surface and doesn't force any
  redesign of the Setup / Control / Operations groupings.
- `guide/codebase_assessment_11may.md` §22 marks **Session
  cloning | ❌ not planned** today. This stub gives it an
  owner without committing to a date.
- 15C's "auto-copy operator library on session create" pattern
  is the natural precedent: per-session copies of library
  RTDs / RuleSets already exist as a category, so the cloning
  service is mostly "do the same thing the operator-library
  auto-copy does, but sourced from a sibling session instead
  of the operator's library".

## Scope (sketch)

### Part 1 — Clone-session service + route

**Goal.** A new `sessions.clone_session(db, source_session,
*, new_name, owner_user, correlation_id) -> ReviewSession`
service helper that creates a fresh session and copies
setup-shape rows from `source_session`.

Likely shape:

- **Copied as setup-shape:** reviewers, reviewees,
  relationships, instruments (with RTD pointers cloned via
  the same library-clone mechanism 15C ships),
  `instrument_display_fields`, `instrument_response_fields`,
  `session_rule_sets`, `session_field_labels`,
  `email_template_overrides`, `responses_received_enabled`,
  `help_contact`, deadline (offset by operator-chosen
  delta or cleared), `email_template` overrides.
- **NOT copied:** `assignments` rows (regenerate from
  rule_set after clone), `responses`, `invitations`,
  `email_outbox`, `audit_events` (the clone target gets a
  fresh `session.created` event with `reason="cloned"` +
  `refs.source_session_id`).
- **Lifecycle state:** clone target lands in `draft`
  regardless of source state. Operator runs Validate +
  Generate + Activate as they would for any fresh session.
- **New session code:** generated by the existing
  `_generate_session_code` helper; the source's code is
  not reused.
- **Audit events:** `session.cloned` on the source
  (`refs.target_session_id`) + the canonical
  `session.created` on the target (`reason="cloned"`).

Likely shape (UI):

- "Clone session" button on the Sessions lobby (per-row
  action) and/or on the Session Home Danger Zone (or a new
  non-destructive "Actions" card). Decide during scoping.
- Clone-session form: prompts for the new session name +
  optionally a deadline override; everything else is copied
  verbatim.

### Part 2 — Cross-operator handoff (post-MVP)

**Goal.** Clone a session **into another operator's
ownership** (or copy it to themself from a colleague's
session).

Likely shape (deferred — confirm need with 16B's role
delegation work):

- The clone target's `creator_id` is the destination
  operator (not the source's owner).
- `SessionOperator` row inserted only for the destination
  operator (the source's operator list is not copied; new
  operators added via 16B's UI).
- Audit-event payload widens to include both actor + target
  operator IDs.

## Hard dependencies

- **None strictly required.** The clone service can land
  even before 15B / 15C ship — the library-clone pattern in
  15C is the cleanest precedent, but the cloning service can
  reproduce the pattern locally if 15C hasn't landed yet.
- **Sequencing hint:** lands more cleanly **after 15C** so
  the library auto-copy helper exists to factor against; and
  **after 15B** so per-instrument RuleSet pointers
  (`instruments.rule_set_id`) have a meaningful "copy this
  pointer to the equivalent target-session row" semantic.
- **Part 2 dependency:** 16B (operator role delegation
  surface).

## Out of scope

- **Cross-deployment / cross-tenant cloning.** Single-deploy
  by design (`codebase_assessment_11may.md` Weakness #10).
- **Cloning responses.** Responses are session-runtime data,
  not setup; clone targets always start with empty
  responses.
- **Copy-paste of partial setup** ("just copy the instruments
  from session A"). Cloning is whole-session; partial copies
  remain a Settings-CSV exercise.
